Atomic CRM ships an *agent harness*: a team of specialized Claude Code agents — planner, developer, reviewer, tester, documenter — that turns a plain-English feature request into planned, tested, reviewed and committed code through a deterministic pipeline. The agents share the same coding guidelines as the Atomic CRM core team, and add tools that cut token usage and speed up development.

## Prerequisites

- The [Claude Code CLI](https://docs.claude.com/en/docs/claude-code) installed and authenticated (an Anthropic API key or a Claude subscription).
- Project dependencies installed (`make install`).
- For full-stack runs only: a local Supabase instance up (`make start-supabase`).

## Usage

Start a Claude Code session as usual. In your prompt, mention the harness to trigger it. For instance:

```
Create a new field for lost deal reasons, and implement it with the harness.

# or

Build a CRM for my company Acme, a recruiting agency, using the harness.
```

The harness starts immediately (or once you exit plan mode): Claude Code will delegate the coordination of the work to a specialized subagent called `chat-orchestrator`, and this agent will spawn other specialized agents depending on the complexity of the task. 

You can also launch the harness from the terminal with a specialized make command. For example:

```sh
make harness REQUEST="Add a reason for a lost deal"
```

:::caution
The harness runs in automatic-permission mode, which means it can execute arbitrary commands on your machine and may modify or delete files. We advise you to run the harness in a sandboxed environment, such as a DevContainer (already configured in the repo) or a VM.
:::

:::tip
The subagents work in a temporary directory (in `/tmp/<repo-slug>`) to keep your main checkout clean. The harness eventually backports the changes to your checkout in a branch and merges it with the current branch.
:::

:::tip
Based on the complexity of the feature request, the orchestrator runs either one of two workflows:

- **Simple**: a single cosmetic edit, a single-field change on an existing entity, or a filter reusing existing components. One developer agent, no planning overhead.
- **Complex**: everything else. The planner decomposes it into tickets; developers, reviewers, and the merger run as a coordinated wave.
:::


## Reverting a Change

Because the merge is automatic, the agent harness may make a change you're not happy with. You can use the following command to revert the last session and return to the state of your branch before the harness started:

```sh
make clean-harness
```

This hard-resets your start branch to the exact commit it was on before the session began.

:::caution
`make clean-harness` is a hard reset: any uncommitted work is stashed first (recover it with `git stash pop`), but committed work made *after* the session started on that branch is also dropped. Run it only when you want to discard the whole session.
:::

## Disabling the Harness

Sometimes Claude Code may use the harness even though you didn't ask for it. If you don't want to use it, you can opt out in your request. For instance:

```
Create a new field for lost deal reasons #no-harness

## or

Create a new field for lost deal reasons without the agent team

## or

Create a new field for lost deal reasons, skipping the agent team
```

The main thread then implements the change itself, no agents, even after a plan is approved. Add *"no-harness for this session"* to keep it off for the whole session.

## Monitoring a Session

The harness writes a detailed trace of every agent, hook, and ticket. Three commands let you follow it:

```sh
make watch                  # live monitor of the most recent session
make monitor                # one-shot summary of the most recent session
make monitor SESSION=<id>   # summary of a specific session
make sessions               # list known sessions, newest first
```

`make watch` re-renders a live view of the active agents, hook activity, and ticket statuses, useful to keep open in a second terminal while a COMPLEX session runs. `make monitor` prints the same information as a single snapshot.

## CRM Builder: A Visual, Ready-Made Environment

[**CRM Builder**](https://github.com/marmelab/crm-builder) wraps the same harness in a containerized, browser-based environment, with no local setup. It adds:

- A containerized environment so you can run the harness without security concerns.
- A pre-installed toolchain, no need to set up Claude Code, Node, or Supabase yourself.
- A live preview of the CRM, updated as the harness applies your changes.
- A session dashboard to follow each run's agents and progress.
- Demo and full-stack mode switching from the UI.
